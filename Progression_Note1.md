## 1. Subjective (S)
**เป้าหมายโครงการ (Project Goal):** 
เพื่อพัฒนาระบบสนับสนุนการตัดสินใจ (Decision Support System) ในการบริหารจัดการอัตรากำลังบุคลากรทางการแพทย์ (Health Workforce) ของเขตสุขภาพที่ 1 โดยเปลี่ยนจากการใช้เพียงข้อมูล "จำนวนประชากร (Population-based)" มาเป็นการใช้ข้อมูล "ภาระโรคที่แท้จริง (Need-based)" และ "ผลลัพธ์ทางการแพทย์ (Outcome-based)" เพื่อหาว่าควรเพิ่มแพทย์เชี่ยวชาญ/พยาบาลสาขาใด ที่โรงพยาบาลไหนอย่างแม่นยำ

**ปัญหาก่อนหน้านี้:** 
ข้อมูลกระจัดกระจาย (HDC, CMI) นำมาวิเคราะห์รวมกันได้ยาก ขาดความเชื่อมโยงระหว่างผลลัพธ์รายโรค (Service Plan) กับภาระงาน จึงเกิดความไม่สมดุลในการกระจายบุคลากร

## 2. Objective (O)
**สถานะปัจจุบันของระบบ (Current System Status):**
ระบบพัฒนาเป็นรูปแบบ Web Application แบบ Interactive (7-Step Wizard) และ Executive Pitch Deck โดยดำเนินการติดตั้งและเชื่อมต่อระบบเรียบร้อยแล้ว

### 2.1 โครงสร้างไฟล์ Frontend (สำหรับนำเสนอและจำลองข้อมูล)
- `pitch_deck.html`: หน้าจอ Executive Presentation สรุปแนวคิด 6 สไลด์ เพื่ออธิบายที่มาที่ไปของ NCO Model
- `simulation.html`: หน้าต่าง User Interface หลักสำหรับจำลองข้อมูล
- `simulation.js`: **Core Engine** ที่ควบคุมการคำนวณ HNI, WCI, Gap และระบบ Recommendation
- `simulation.css`: ชุดคำสั่งตกแต่ง UI (Premium Dark Theme)

### 2.2 โครงสร้างข้อมูลและหลังบ้าน (Backend & Data Sources)
- **`data_inputV1.csv`**: เป็นฐานข้อมูลหลักที่เก็บข้อมูลประชากร, อัตรากำลัง, ข้อมูลสถิติของโรงพยาบาล และค่า Performance
- **ระบบ Web Scraper (ETL - Extract/Transform/Load):**
  1. **CMI Scraper (`fetch_hr.py`)**: ดึงข้อมูลจากระบบรายงานข้อมูล CMI ของ รพ.แม่วาง (cmi.maewanghospital.go.th) เพื่อนำค่า Performance D01-D06 และผลลัพธ์ Service Plan (A-H) รายสาขามาใช้ โครงสร้างข้อมูลเป็นตารางตายตัว สามารถดึงด้วย Request ปกติได้
  2. **HDC Scraper (`scrape_hdc_province.py`)**: ดึงข้อมูลระบาดวิทยา (Epidemiology) จากเว็บ HDC (moph.go.th) *ขอบเขตข้อมูล:* ดึงอัตราป่วยต่อแสนประชากรระดับเขต/จังหวัด (เนื่องจากระดับ รพ. มีความลึกและต้องรอโหลดนาน) ครอบคลุม 4 โรคหลัก: หลอดเลือด/หัวใจ (CVD), มะเร็ง, เบาหวาน (DM), ไตวาย (CKD)

### 2.3 สิ่งที่ดำเนินการไปแล้วใน Simulation (พร้อมตัวอย่าง รพศ./รพท.)
- **ระบบเลือกโรงพยาบาล (Step 0-1):** ผู้ใช้สามารถเลือก รพศ. (เช่น รพ.ลำปาง, นครพิงค์) หรือ รพท. (เช่น น่าน, ลำพูน) เพื่อดูข้อมูลเจาะลึก พร้อมระบบแจ้งเตือนคุณภาพข้อมูล (Data Quality วินิจฉัยผิดพลาด ป่วยนอก/ใน)
- **รวมสถิติระดับประเทศเป็นอ้างอิง:** ฝังข้อมูลความสูญเสียปีสุขภาวะ (DALY 2019) แบบแยกเพศและช่วงอายุ เป็น Tabbed UI ซ้อนไว้ในระบบเพื่อให้ผู้บริหารดูอ้างอิง
- **แสดงข้อมูลระบาดวิทยาตามบริบทพื้นที่:** เช่น หากเลือก รพ.แพร่ ระบบจะดึงฐานความชุก 4 โรคหลักของ จ.แพร่ มาแสดงผลเพื่อใช้อธิบาย "ความหนัก" ของเคสประจำจังหวัด

## 3. Assessment (A)
**หลักการคิดแต่ละขั้นตอนของ Core Engine (Algorithms & Calculations):**

### 3.1 การคำนวณความต้องการทางสุขภาพ (Health Need Index - HNI)
*คำถาม: พื้นที่นี้มีความต้องการพึ่งพาระบบสาธารณสุขมากหรือน้อยเพียงใด?*
- **ตัวแปรหลัก:** 
  1. สัดส่วนผู้สูงอายุ (Elderly)
  2. อัตราความชุกโรคเรื้อรัง (Chronic) 👉 *คำนวณโดยนำ Prevalence Rate (ต่อแสนคน) ของ 4 โรค [CVD + Cancer + DM + CKD] มาบวกกัน*
  3. ความเสี่ยงสุขภาพจิต (Mental)
- **วิธีการคิด (Dynamic Normalization):**
  นำค่าดิบของ รพ. ที่เลือก ไปหารด้วย "ค่าสูงสุดของกลุ่มโรงพยาบาลในเขต 1 ทั้งหมด" (Max Value in Region) เพื่อแปลงเป็น Base Score 0-1.0 จากนั้นนำไปถ่วงน้ำหนัก (Weight) ที่ผู้ใช้หมุน Slider ปรับแต่งได้ (เช่น 40:40:20) แล้วคูณ 100 
  *สูตร:* `Score = (Σ Weight_i * (Raw / Max_in_Region)) / Total_Weight * 100`
  *ผลลัพธ์:* ได้คะแนน 0-100 (แดง > 70, เขียว < 50)

### 3.2 การประเมินกำลังคน (Workforce Capacity Index - WCI)
*คำถาม: โรงพยาบาลนี้มีกำลังคนเพียงพอต่อขนาดประชากรและบริบทหรือไม่?*
- **คำจำกัดความ (Query Definitions):** 
  - **แพทย์ทั้งหมด:** นับจากกลุ่มอาชีพ **"นายแพทย์"**
  - **พยาบาลทั้งหมด:** นับจากกลุ่มอาชีพ **"พยาบาลวิชาชีพ"** และ **"พยาบาลวิชาชีพ/นักวิชาการสาธารณสุข"**
- **วิธีการคิด:** แปลงผู้ให้บริการ (แพทย์/พยาบาล/เภสัช) เป็นอัตราส่วน "คน ต่อ ประชากร 10,000 คน" และนำไปเทียบกับค่ามัธยฐานฐาน (Median Standard) ที่ตั้งไว้ (เช่น หมอ 3.5 คน, พยาบาล 12 คน / หมื่นปชก.) จากนั้นแปลงออกมาเป็นคะแนนเต็ม 100

### 3.3 การวิเคราะห์จุดวิกฤติ (Gap Analysis)
*คำถาม: คนพอเหนื่อยไหม เคสล้นมือไหม?*
- **วิธีการคิด:** `GAP = HNI Score - WCI Score`
- **การแปลผล:** ค่าบวก (+) แปลว่า ภาระงาน (Need) นำหน้ากำลังคน (Capacity) ขาดความสมดุล (Overworked)

### 3.4 การจับคู่เพื่อยืนยันปัญหา (Decision Matrix: Gap × Outcome)
*คำถาม: เหนื่อยแล้วผลงานออกมาดีไหม หรือคนพอแต่ผลงานแย่?*
ระบบจะนำ GAP มาไขว้กับค่า Performance Outcome (อัตราตายต่างๆ จาก Step 5) เพื่อวิเคราะห์รากปัญหาตัวจริง:
1. **🔴 วิกฤติ (Gap สูง + Outcome แย่):** ภาระงานล้นมือ ทำให้คนไข้เสียชีวิตเยอะ 👉 *ต้องเพิ่มคนด่วน ควบคู่กับปรับระบบ*
2. **⚡ ระวัง (Gap สูง + Outcome ดี):** ประคองตัวได้แต่เหนื่อยมาก ทิ้งไว้นานคุณภาพจะตก 👉 *เพิ่มคนเชิงป้องกัน Burnout*
3. **🟡 ปัญหา Process (Gap ต่ำ + Outcome แย่):** คนเพียงพอแต่คนไข้ยังตายเยอะ 👉 *ปัญหาไม่ได้อยู่ที่ขาดคน แต่อยู่ที่ Protocol / เครื่องมือมือ / ทักษะ (Skill Mix)*
4. **🟢 ดี (Gap ต่ำ + Outcome ดี):** สมดุลดี 👉 *ถอดบทเรียนเป็น Best Practice*

### 3.5 การแนะนำอัตรากำลังเฉพาะทาง (Service Plan Specialty Mapping)
*คำถาม: ถ้าต้องเติมคน ต้องเติมให้สาขาไหน อาชีพใด?*
ระบบใช้ตัวชี้วัดความตาย/ความสำเร็จเฉพาโรคมาเป็นเกณฑ์ตัด (Threshold):
- *สูตร:* หาก `Indicator_Value > Threshold` (ในกรณีที่น้อย=ดี เช่น อัตราตาย) ระบบจะแสดงคำแนะนำทันที
- **ตัวอย่างการจับคู่ (Mapping):**
  - **STEMI Mortality > 12%** 👉 แนะนำให้เติม: `อายุรแพทย์หัวใจ Interventional` และ `พยาบาล CCU/Cath Lab`
  - **Stroke Mortality > 15%** 👉 แนะนำให้เติม: `อายุรแพทย์ประสาท` และ `พยาบาล Stroke Unit`
  - **มะเร็ง Mortality > 15%** 👉 แนะนำให้เติม: `อายุรแพทย์มะเร็ง` และ `พยาบาลเคมีบำบัด`
- *หลักการ:* ดึงความต้องการจาก "คนไข้ที่เสียชีวิตในระบบจริงๆ" มาตอบว่าขาดแพทย์ความเชี่ยวชาญสาขาใด ไม่ใช่นั่งเทียนเดาเอาจากจำนวนเตียง

## 4. Plan (P)
**แผนการดำเนินงานและการส่งมอบทีม IT (Technical Hand-off & Next Steps):**

### 4.1 การจัดการโครงสร้างข้อมูล (Data Infrastructure)
- **Current Technical Debt:** โปรแกรมโหลดไฟล์ CSV มา Parse ฝั่งหน้าบ้าน (Client-side) ซึ่งไม่ปลอดภัยและช้าเมื่อขนาดข่อมูลใหญ่มาก
- **IT Task:** ย้ายระบบฐานข้อมูลจาก `.csv` ไปเป็นโครงสร้าง Relational Database (SQL) หรือ NoSQL (MongoDB/Firebase) และสร้าง RESTful API หรือ GraphQL เพื่อให้ `simulation.js` เรียก `fetch('/api/hospitals')` แทน

### 4.2 ระบบดึงข้อมูลอัตโนมัติ (Automated ETL Pipelines)
- **Current State:** มีสคริปต์ Python ต้นแบบ `fetch_hr.py` และ `scrape_hdc_province.py` 
- **IT Task:** นำ Script เหล่านี้ไปขึ้น Server จัดทำขบวนการทำงานอัตโนมัติ (Cron Jobs) ดึงข้อมูลอัปเดตทุกเดือน
- *ข้อพึงระวัง:* ระบบ HDC มีการปรับเปลี่ยน UI (Dynamic SPA) และมี Popup หาก Scraper รันไม่ผ่าน ทีม IT ต้องปรับแก้ DOM Selectors หรือพิจารณาขอ API จาก HDC โดยตรง (System-to-System Integration)

### 4.3 สิ่งที่ต้องพัฒนาเพิ่มในอนาคต (Future Implementations)
1. **Medical Council Registry (แพทยสภา):** ปัจจุบันเรามีแค่ยอดรวมของแพทย์ ทีม IT ควรทำ Data Linkage กับแพทยสภา เพื่อให้ทราบ "จำนวนแพทย์เฉพาะทางย่อย (Sub-specialty)" จริงๆ ที่อยู่ใน รพ. แผนกนั้น เพื่อจะหักลบกับ Need ได้แม่นยำขึ้น
2. **PM 2.5 & Environmental Factors:** นำข้อมูลความหนาแน่นฝุ่นควันมาเป็นอีกหนึ่งตัวแปร (Slider) ผูกสมการเข้ากับอัตราป่วยมะเร็งปอด / COPD
3. **FTE (Full-time Equivalent) Algorithm:** ปรับวิธีคำนวณ WCI (Step 3) จาก Headcount เป็นการคิดเนื้องาน FTE แยกตามขนาดเตียง
4. **Interactive Dashboard Overview:** หน้าสรุปรวม (War Room) ให้ผู้ตรวจราชการเห็น Gap กองรวมกันทั้ง 8 จังหวัด เพื่ออนุมัติตำแหน่งได้ในคลิกเดียว
