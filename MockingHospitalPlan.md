# Mocking Hospital Plan
**Project:** NCO Simulation - Sandbox Mode
**Concept:** "Create-a-Character" for Hospitals (ระบบสร้างตัวละครโรงพยาบาลสมมติ)
**Objective:** อนุญาตให้ผู้ใช้งานสร้างโรงพยาบาลใหม่ขึ้นมาเองทั้งหมด (Sandbox) โดยสามารถกำหนดจำนวนประชากร ระบาดวิทยา และที่สำคัญที่สุดคือ "อัตรากำลังแพทย์/พยาบาลเฉพาะทาง (Sub-specialty)" เพื่อจำลองการจัดสรรและทดสอบกระบวนการคิดของ NCO Model

---

## 1. System Overview (ภาพรวมระบบ)
ระบบ Mocking Hospital จะทำงานคู่ขนานกับระบบ Simulation ขั้นพื้นฐาน โดยเปลี่ยนจาก "การเลือกโรงพยาบาลที่มีอยู่แล้วในระบบ" มาเป็น "การสร้างโรงพยาบาลใหม่ (Canvas)" เสมือนการสร้างตัวละครในเกมแนว RPG (Role-Playing Game) 
ผู้ใช้จะต้องจัดสรรตัวเลขต่างๆ ตั้งแต่ระดับพื้นฐาน ไปจนถึงระดับลงลึก (Sub-specialty) เพื่ออุดช่องโหว่ด้านข้อมูลความเชี่ยวชาญเฉพาะทางที่ระบบปัจจุบันไม่มี

---

## 2. Sandbox Workflow (ขั้นตอนการเข้าสู่ระบบจำลอง)

### ขั้นตอนที่ 1: Basic Profile (สร้างกรอบโครงสร้าง)
ผู้ใช้งานจะเป็นผู้กำหนดข้อมูลตั้งต้นของโรงพยาบาล:
- **ชื่อโรงพยาบาล / รหัส:** (Text Input)
- **จังหวัด:** (Dropdown) เพื่อให้ระบบดึงค่าระบาดวิทยาประจำจังหวัด (Provincial Prevalence) มาตั้งเป็นแกนกลางอัตโนมัติ
- **ระดับโรงพยาบาล:** A (รพศ.), S (รพท.), M1, F (รพช.) เพื่อกำหนดค่า Median อ้างอิงและ Service Plan ระดับพื้นที่
- **จำนวนเตียง:** (Number Input) เพื่อคำนวณอัตราภาระงานต่อเตียง
- **กลุ่มประชากรที่ต้องดูแล (Type 1,3):** (Number Input) สังกัดประชากรหลัก

### ขั้นตอนที่ 2: Health Needs & Epidemiology (กำหนดภาระโรค)
- **Default Load:** ระบบจะ Default สัดส่วนมาจากข้อมูล HDC จริงของจังหวัดที่เลือกในขั้นที่ 1 (เช่น ผู้สูงอายุ 24%, ค่าบวกโรคเรื้อรัง 4 โรค 19,000 ต่อแสนคน)
- **Override (Sandbox Feature):** ผู้ใช้งานสามารถ "ปรับแก้ (Override)" ตัวเลขเหล่านี้ได้ด้วยมือ เพื่อจำลองภาพสมมติ (เช่น หากมีการระบาดของโรคหลอดเลือดสมองเฉียบพลันเพิ่มขึ้น 2 เท่า จะเกิดอะไรขึ้น)

### ขั้นตอนที่ 3: General Workforce (บุคลากรภาพรวม)
กรอกตัวเลขบุคลากรพื้นฐานก้อนใหญ่:
- แพทย์ภาพรวม (นายแพทย์/GP)
- พยาบาลวิชาชีพภาพรวม
- เภสัชกรและสหวิชาชีพอื่นๆ

### ขั้นตอนที่ 4: Sub-specialty Allocation (จัดสรรความเชี่ยวชาญเฉพาะทาง - จุดสำคัญ!)
นี่คือ Core Feature ใหม่ของระบบ ผู้ใช้ที่มีโควตา "แพทย์/พยาบาล" จะสามารถแตกลง Detail ได้ดังนี้:

**หมวดอายุรศาสตร์และหลอดเลือด:**
- อายุรแพทย์หัวใจ (Cardiologist) / แพทย์ทำหัตถการ (Interventional): __ คน
- อายุรแพทย์ประสาทวิทยา (Neurologist): __ คน
- พยาบาลรับฝีก CCU / Cath Lab: __ คน
- พยาบาลคลินิก Stroke Unit / Fast Track: __ คน

**หมวดศัลยกรรมและอุบัติเหตุ:**
- ศัลยแพทย์ทั่วไป: __ คน
- ศัลยแพทย์สมองและระบบประสาท (Neurosurgeon): __ คน
- พยาบาล Trauma / OR: __ คน

**หมวดเฉพาะทางทารกและมารดา:**
- สูตินรีแพทย์: __ คน
- กุมารแพทย์ทารกแรกเกิด (Neonatologist): __ คน
- พยาบาล NICU / ห้องคลอด: __ คน

*หมายเหตุ: ข้อมูลในหน้านี้จะไป Plug-in เข้ากับโมดูล Service Plan ใน NCO Model*

### ขั้นตอนที่ 5: Performance & Service Plan Simulator (จำลองผลลัพธ์การทำงาน)
ผู้ใช้กำหนดค่า "ผลสอบ" (Performance Outcome) ที่โรงพยาบาลสมมตินี้ทำได้เพื่อใช้ไขว้แกน Gap:
- Case Mix Index (CMI) (ความยากง่ายของโรคเฉลี่ย)
- อัตราตาย STEMI Mortality (%)
- อัตราตาย Ischemic Stroke (%)
- อัตราตาย Sepsis (%)
- ฯลฯ

## 3. Integration with NCO Core Engine (การประมวลผลคำแนะนำ)
เมื่อกรอกเสร็จครบทั้ง 5 ฟอร์ม ระบบจะนำข้อมูลเข้าสู่กระบวนการทั้ง 7 Step ของ NCO Simulation ปกติ แต่ความแม่นยำและการเจาะลึกจะเหนือกว่าแบบดั้งเดิม:

1. **Specific WCI (Workforce Capacity แยกส่วน):** แทนที่จะบอกแค่ว่า "หมอรวมมีพอหรือไม่พอ" ระบบจะเจาะจงได้ว่า "จำนวนแพทย์รวมพอ แต่ขาดอัตรากำลัง Neurologist ขาดแคลนอย่างหนักเมื่อเทียบกับขนาดประชากรสูงอายุ"
2. **Deep Gap Analysis:** เอา Need เฉพาะด้าน (เช่น อัตราป่วย Stroke สูงในจังหวัด) มาหักลบชนกับ Number of Neurologist โดยตรง
3. **Smart Decision Matrix (ตัวอย่างผลลัพธ์แบบเจาะลึก):**
   - *ตัวอย่างที่ 1:* หากตั้งค่า "STEMI Mortality = 20%" แต่อัตราป่วย CVD ของจังหวัดสูงปรี๊ด และในหน้า Sub-specialty ใส่เลข "อายุรแพทย์ Interventional = 0"
     👉 **ระบบสรุป (AI Recommendation):** "วิกฤติรุนแรง: ความชุกของโรคหัวใจสูง แต่อัตราตายทะลุเกณฑ์เนื่องจากไม่มีแพทย์เฉพาะทาง (Interventional Cardiologist) เสนอให้จัดสรรโควตาเร่งด่วน 1-2 ตำแหน่ง พร้อมส่งพยาบาลอบรม CCU เพิ่ม"
   - *ตัวอย่างที่ 2:* หากตั้งค่า "อัตราตาย Sepsis = 45%" แต่ในหน้า Sub-specialty ใส่พยาบาล ICU เต็มความจุแล้ว
     👉 **ระบบสรุป:** "Capacity ด้านบุคลากรเพียงพอ แต่ Outcome ยังวิกฤติ สะท้อนปัญหาการคัดกรอง Sepsis Fast Track หรือข้อจำกัดด้านเครื่องมือแพทย์ ไม่ควรแก้ปัญหาด้วยการเพิ่มพยาบาล ICU อย่างเดียว"

## 4. Technical Implementation Path (แนวทางการเขียนโค้ดและพัฒนา)
1. **State Management:** 
   - สร้าง JSON Blueprint `MockHospitalState` เพื่อบันทึกค่าทุก field ที่ผู้ใช้กรอก
2. **UI Component (`mock_builder.html` หรือ Modal):**
   - ใช้หลักการ Wizard / Accordion Form ให้ผู้ใช้กาง/พับ กรอกข้อมูลทีละหมวด (1. Profile ➔ 2. Need ➔ 3. Sub-spec ➔ 4. Outcome)
   - มีปุ่ม "Fill with Defaults" เพื่อให้ผู้ใช้ไม่ต้องเหนื่อยพิมพ์ศูนย์ทุกช่อง
3. **Core Engine Upgrade (`simulation.js`):**
   - Refactor ฟังก์ชัน `getIndicatorValue()` และ `HOSPITALS[name]` ให้สามารถรับพารามิเตอร์แบบ "Mock Object" นอกเหนือจากการดึง CSV
   - ปรับปรุงตาราง `SPECIALTY_MAP` ให้เช็คชนกับตัวแปรจากฟอร์ม Sub-specialty ของ Mock_State ได้ตรงๆ (ตอนนี้ใช้ mapping กว้างๆ)
4. **Data Persistence:**
   - ใช้ LocalStorage ในการ Cache หน้าฟอร์ม กรณีผู้ใช้กรอกค้างไว้แล้วหน้าเว็บ Refresh
   - ทำฟีเจอร์ "Export/Import Scenario (.json)" เพื่อให้ผู้ใช้ส่งสถานการณ์จำลองที่ตั้งค่าไว้ไปให้ทีมงานอื่นรันดูผลลัพธ์ได้
